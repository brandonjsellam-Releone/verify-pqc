# PQ SDK — Security Review & Fixes (11-seat council, 25 Jun 2026)

The three reference modules (polarseek, pqsign, pqef) were adversarially reviewed by 6 council seats (DeepSeek, NVIDIA/Nemotron, Moonshot/Kimi, Grok, OpenAI, Gemini-verify). Each module had one real, high-confidence bug. All fixed; each fix has a regression test encoding the attack. **All module self-tests pass** (`node test-all.mjs` → ALL MODULES PASS).

> **As-of note (counts are point-in-time):** This is a *chronological review log*; the quantitative figures in each entry below record what was true at that round and are deliberately left frozen. After later hardening sweeps the live harness counts are now higher — **tamper-binding 1,064 assertions (now covers ALL 12 of this session's signed cores — 5 platform engines + 7 Wave-2), fuzz 60×72 = 4,320 calls (0 fail-open observed), 65 distinct domain-separation contexts, canon byte-identical across 28 modules; + the Wave-2 cores (pqshield/pqcap/pqadmit/pqconsent/pqpay/pqvc/pqfirmware) + the platform engines `pqmonitor` (SOC) / `pqgate` (admission) / `pqflow` (payment lifecycle) / `pqdelegate` (agent delegation — council-reviewed: scope/max_uses widening gaps found by DeepSeek+OpenAI, fixed to narrow-only) / `pqconsentflow` (consent lifecycle — recompute-verified permitted-state + access-after-withdrawal violation detection) with `conformance-vectors.mjs` (25 KAT).** The current authoritative inventory is `SIGNED_OBJECTS.md` / `AUDIT_DOSSIER.md` / `domain-separation.mjs` output — not the historical numbers in this log.

## Bugs found & fixed
1. **pqsign — caller-supplied Merkle leaf (CRITICAL).** `verifyBundle` checked an inclusion proof against a *caller-supplied* `leaf` without confirming `leaf == leafHash(att)`. A valid signed attestation could be paired with a *different* logged entry's proof and still verify. **Fix:** `verifyBundle` now recomputes the leaf from `att` and rejects a mismatch (`leafBound`). Test: "valid att + another leaf's proof → FAILS."
2. **PQEF — self-attesting trust (CRITICAL).** The verifier trusted the `public_key_hex` + `validated_module` flag *from the bundle*, so a self-signed bundle could claim `COMPLIANCE-VERIFIED`. **Fix:** `COMPLIANCE-VERIFIED` now requires the signing key to match a **pinned `opts.trustedIssuers`** supplied by the verifier; otherwise the verdict downgrades to "SIGNED (untrusted issuer…)". `validated_module` is treated as a *claim*, not a fact. Test: "self-signed bundle, no trusted issuer → NOT compliance-verified."
3. **POLARSEEK — non-canonical envelope hash.** The custody record's `envelope_hash` used `JSON.stringify` (key-order fragile; and would silently drop binary fields if they weren't hex). **Fix:** canonical fixed-order concatenation of the hex fields (`envelopeHash`), plus `verifyCustodyBinding(envelope, record)`. Tests: binding verifies; tampered envelope → binding FAILS. (Note: AES-GCM tags were already the integrity mechanism; this hardens the provenance record.)

## How to use the SDK safely (the council's standing guidance)
- **PQEF / pqsign verifiers MUST be given a pinned trust root** (`trustedIssuers` / known signer + log keys) out-of-band. A valid signature under a *bundle-supplied* key proves only "signed by that key," never "signed by an authorized issuer." Compliance verdicts without a pinned root are meaningless.
- **Canonical encoding:** v0.1 reference uses canonical-JSON (JCS-style) / fixed-order concat. The **production profile is deterministic CBOR (RFC 8949)**; signer and verifier MUST use the *same* profile. Never sign raw `JSON.stringify` across implementations.
- **Replay / freshness:** these primitives don't enforce it. Apps MUST add timestamp/nonce windows + monotonic STH `tree_size`/`ts` (anti-rollback) + gossip for equivocation detection.
- **POLARSEEK is KMS custody, not messaging FS:** sealing to a long-term custody key is by design (secrets must be unsealable later); the ML-KEM leg gives HNDL protection of the wrapped key.
- **PQRS** should move to deterministic integer arithmetic (basis points, tolerance 0) for the production verifier (Gemini); the v0.1 float+0.15-tolerance is reference-only.
- **Still NOT a FIPS-140-3 validated module** — these use FIPS-203/204 *algorithms* via @noble; constant-time/side-channel guarantees are implementation-dependent (JS can't guarantee constant-time). Falcon = on-chain/provenance only (draft FIPS 206).

## pqtransport handshake — 3-seat review (DeepSeek, NVIDIA, Moonshot), all FIX-FIRST → fixed (9/9)
The hybrid handshake had the classic SIGMA gaps; all fixed + regression-tested:
- **Identity not bound into the transcript → unknown-key-share.** Fix: both identity pubkeys are hashed into `th` (initiator now sends its identity in msg1). Pinning becomes defence-in-depth, not the sole binding.
- **Same transcript + same signing context for both roles → reflection/role-confusion.** Fix: initiator and responder sign under DISTINCT contexts (`…/initiator`, `…/responder`); a regression test confirms the responder's signature does NOT verify under the initiator context.
- **Random GCM nonces over a session → birthday-bound reuse (catastrophic).** Fix: deterministic per-direction COUNTER nonces + seq in AAD; separate i2r/r2i keys.
- Honest labeling: HYBRID PQ/classical (X25519 is classical; PQ confidentiality rests on ML-KEM-1024). Deferred: encrypt the initiator identity for privacy; an explicit key-confirmation message + a strict "no key use before msg3 verified" state machine.

## pqanswer notary — 2-seat review (DeepSeek, Moonshot), FIX-FIRST → fixed (9/9)
- **Misleading "verified" on selective disclosure.** Hashes-only verification could read as fully verified. Fix: split into `binding_verified` (authentic dossier from a trusted signer) vs `content_verified` (sources supplied + matched); top-level `verified` = content_verified (strict).
- **Source order / duplicate-URL ambiguity.** Fix: canonical source sort by (url, content_sha256) at notarize + order/duplicate-safe **set** comparison at verify (regression test: reordered sources still verify).
- **Replay.** Fix: optional verifier **nonce** bound into the signed dossier; `expectedNonce` mismatch → not verified.
- **Provenance ≠ support (the key honesty fix).** The note now states plainly it attests only that the answer was generated for this query using these source snapshots by this signer — NOT relevance, support, completeness, or truth (sources may be genuine yet not support the answer).
- **Deferred to v0.2:** per-claim→chunk attribution (claim_id + span + relationship tag) to move from "provenance" toward "auditable support"; a retrieval-scope/corpus-version commitment (defeats selective omission); signer key-governance (rotation/revocation/threshold) so "trusted signer" isn't a single point of trust; a verifier UX that resists being read as a "truth seal."

## pqinduct induction (Cycle I) — 4-seat review (DeepSeek/Grok/Mistral/NVIDIA), all FIX-FIRST → fixed (16/16)
The Order of the Lemniscate's cryptographic-merit induction. Convergent findings, all addressed + regression-tested:
- **Shareable nonce falsified the merit claim (Grok/NVIDIA, CRITICAL) → fixed.** A public cycle nonce meant the credential only proved "someone solved it + I can run ML-DSA." Now the inner ring binds `solution_binding = H(cipher_key ‖ solver_pubkey)`: the solver must POSSESS the solution secret AND bind it to their OWN key. Regression: a different key with only the shared nonce → INVALID. (Full anti-share/Sybil = per-solver assignment nonce + ZK-personhood nullifier = deferred gated registration; Cycle I proves OPERATIONAL competence, not cryptanalysis — later cycles raise difficulty.)
- **canonicalize ambiguity (DeepSeek) → hardened + documented.** Fail-closed on non-finite numbers + `undefined`; production profile = RFC 8785 / deterministic CBOR (signer+verifier share one). Internally consistent already; this closes cross-impl drift.
- **Equivocation (NVIDIA) → `manifestId()` anchor hook.** The Order could serve different manifests; now `manifestId` = hash of the signed core, anchored publicly before release; verifiers pin `opts.expectedManifestId`. (The public bulletin/log commitment itself = operational.)
- **Credential bound only to a root, not proven-included (NVIDIA) → fixed.** The credential now CARRIES the Merkle inclusion proof; `verifyCredential` recomputes the leaf from the entry and verifies it against the signed `sth_root` (self-contained third-party verifiability). Regression: tampered evidence → `inclusionOk` false. STH root must still come from a witnessed/threshold-signed, gossiped log (operational).
- **One-credential-per-identity (NVIDIA) → issuance gate.** `opts.seenSubjects` rejects a repeat subject key at issuance (a registry/ZK-nullifier is the scalable form).
- **Legal firewall must be substantive, not a label (Mistral) → committed.** The credential commits `not_a_security` + `no_economic_or_service_access` + a `terms` string; `verifyCredential` enforces the firewall (`econOk`). DO/DON'T: NO economic/service/priority access (else MiCA utility-token + FINMA *Anlagetoken*); keep it on the cultural layer with an absolute economic firewall + named council; counsel + a FINMA no-action view before launch; GDPR — the log holds ONLY hashes + a pseudonymous handle (no PII), never link an off-chain identity to a leaf; structural question (issue via a Swiss Verein vs the Stiftung) = legal/owner.
Dual-signature fail-closed (ML-DSA-87 + SLH-DSA-256s) + the no-caller-supplied-leaf inclusion check were confirmed sound. Crypto/protocol findings are fixed in code; the witnessed-STH source, public manifest-commitment, ZK-personhood/assignment-nonce, on-chain ERC-5192 mint, and the legal structure are the deferred operational/legal layer.

## FIPS 203/204/205 conformance (from the corpus deep-read) — TESTED in `fips-conformance.mjs` (12/12)
The full-FIPS-body deep-read produced corrections; we verified each holds on our actual @noble usage:
- **Hedged ML-DSA signing is the default (FIPS 204 §3.4)** — two signatures of the same message DIFFER and both verify (we never set a zero/deterministic `rnd`; @noble is hedged). Deterministic signing is avoided where side-channels matter (§3.4).
- **Context strings (FIPS 204)** — we use the standardized `{context}` parameter (NOT an ad-hoc message prefix); a sig under a context fails without it (tested).
- **Strict length rejection (FIPS 204 §3.6.2)** — wrong-length sig/pk → verify returns `false` (our universal try/catch→false pattern; never accepted, never an uncaught throw).
- **ML-KEM-1024 implicit rejection (FIPS 203)** — a corrupted ciphertext does NOT throw; decapsulation returns a different 32-byte pseudorandom ss (constant-time implicit rejection), tested.
- **Exact sizes** — ML-DSA-87 pk 2592 / sk 4896 / sig 4627; ML-KEM-1024 ek 1568 / dk 3168 / ct 1568 / ss 32; **SLH-DSA-SHAKE-256s** pk 64 / sig 29792.
- **SLH-DSA leg → SHAKE variant (apex-consistency upgrade, deep-read FIPS 205):** switched `slh_dsa_sha2_256s` → `slh_dsa_shake_256s` across pqinduct/pqef/fips-conformance to align with our Keccak/SHAKE stack. Note: SLH-DSA is NOT message-bound under EUF-CMA (FIPS 205 §11) — never assume exclusive-ownership/non-resignability; it's an assumption-DIVERSITY leg (Cat-5, same level as ML-DSA-87), never "the strongest."
- **Hybrid claim precision (deep-read FIPS 203 §3.3):** never claim hybrid security "follows from ML-KEM being IND-CCA2" — the combiner is argued separately (our transcript-bound X-Wing is stronger than baseline concat-into-KDF). ML-KEM ss is not ct-bound (FO removed the ct-hash).
- **No-mix / HashML-DSA pre-hash** — N/A: we use pure ML-DSA only (no HashML-DSA), so the pure-vs-prehash no-mix rule and the ≥2λ pre-hash-digest guard don't apply; if HashML-DSA is ever added, gate each key to one mode + reject <512-bit pre-hash for ML-DSA-87.
Still implementation-dependent (JS): constant-time guarantees — Falcon/provenance leg + the production signing core remain out-of-scope for FIPS-140-3 validation.
- **Deterministic KNOWN-ANSWER vectors (`kat-conformance.mjs`, 10/10):** seed-pinned keygen + deterministic sign (fixed extraEntropy) + seeded encaps, frozen to expected SHA-256 digests (+ the ML-KEM shared secret in full) for ML-KEM-1024 / ML-DSA-87 / SLH-DSA-SHAKE-256s. Catches any @noble version drift / regression. NOT the official NIST ACVP vectors — those are an owner drop-in (usnistgov/ACVP-Server JSON); this pins our own engine's deterministic output for reproducibility.

## pqratchet-he — header-encryption / sealed-sender (apex messaging variant) — DeepSeek FIX-FIRST → reconciled (10/10)
The metadata-private Triple Ratchet (Signal HE): root KDF emits (RK,CK,NHK); headers AEAD-encrypted under rotating header keys (HKr/NHKr), so the wire reveals no keys/counters/ratchet-signal. Supersedes pqratchet for QuantumMesh production. DeepSeek raised 4 points; verified against the HE spec, 2 do NOT apply (applying them would regress):
- **skipped-keys-first order = INTENTIONAL/spec-correct** (a current-chain OOO message's header also decrypts under HKr → "HKr-first" mis-routes it). O(MKSKIPPED) trial decrypts, BOUNDED by MAX_SKIP (standard Signal-HE DoS mitigation). Kept; noted a production fast-path.
- **random header nonce = ON PURPOSE** (a plaintext counter leaks message index / chain length — the exact metadata HE hides). HKs rotates per chain → 96-bit random birthday risk unreachable. Kept random; DeepSeek's "use counter" rejected as privacy-harmful.
- **HKDF domain separation present** (root vs hka vs nhkb use distinct info) — confirmed, no change.
- **payload AAD binds the encrypted-header bytes** — sound (DeepSeek agreed).
Lesson logged: verify review fixes against the spec; don't cargo-cult — 2/4 here were wrong for the HE setting.

## pqkt — Key Transparency log + monitor (NEW module) — TWO council rounds (Grok → OpenAI) → hardened (pqkt 15/15)
Append-only ML-DSA-87 Merkle log of issuer key bind/revoke + a monitor. Built on pqsign's RFC-6962 inclusion (index/tree_size-bound) + NEW **consistency proofs** (`verifyConsistency`, §2.1.2 — Gemini-confirmed faithful; pqsign 17/17). Council found real holes, fixed before acceptance:
- **Grok: no issuer authorization → operator could mint keys → WEAKER than pinning.** FIXED with CONIKS-style authority chaining: bind/revoke signed by the issuer's current key; `resolveIssuerKey` ignores unauthorized events → a log operator cannot bind a key for an issuer it doesn't control.
- **OpenAI (4 findings):** (1) **replay/rollback** w/o monotonic seq → added per-issuer **monotonic seq**, exact-next-seq, first-wins on branches; (2) **TOFU-as-default** → bootstrap now requires `expectedBootstrap` pin OR explicit `allowTofu` (off by default); (3) **post-revoke rebind** (CRITICAL — revoke→null then attacker self-bootstrap seizes issuer) → **UNSEEN→ACTIVE→REVOKED** state machine, REVOKED terminal; (4) **shared signing context** → distinct AUTH vs POSSESSION contexts. All five have explicit attack tests.
- HONEST LIMITS (documented): completeness / client-specific-view (a single log can omit events) needs gossip/witnesses; resolveIssuerKey assumes inclusion/STH-verified, log-ordered input; re-enrollment after revoke is out-of-band.
- **WITNESS / GOSSIP layer (Hermes review) — pqkt 24/24:** added `Witness.cosign` (verify + co-sign (log_pub,size,root); refuse a conflicting same-size root = local non-equivocation), `verifyWitnessedSTH` (k-of-n distinct trusted witnesses), and `gossipDetectEquivocation` (aggregates self-describing cosigs and flags any tree_size attested with two distinct roots). Hermes correctly noted k-of-n alone does NOT close split-view — a log can PARTITION witnesses; the gossip detector catches that (tested: a fresh witness co-signs the fork, gossip flags the conflict). Honest scope now states GOVERNANCE (log controls ≤k-1 of n), DURABILITY (`seen` must persist), and that the residual is running INDEPENDENT witnesses + a real gossip channel. Reframed "split-view resistant" → "NARROWS / detectable-with-gossip".

## pqconsentflow consent-lifecycle red-team (30 Jun) — DeepSeek (5th platform engine)
The VaultHealth consent-LIFECYCLE engine: a controller-signed, hash-chained grant→access→withdraw log over pqconsent; `verifyConsentFlow` recomputes the permitted state and flags violations, returning `verified` (chain authentic + well-formed) separately from `compliant` (no violation evidenced). DeepSeek adversarial pass — **3 real findings, all fixed + re-greened** (61-module gate; pqconsentflow 19/19; tamper-binding +1 scenario):
- **"hide a post-withdrawal access by back-dating `at` / reordering the chain" — REAL, FIXED.** The first cut trusted chain position + the controller-chosen entry `at`. `foldState` now ALSO compares each access against the SUBJECT-signed `revoked_at` (from the embedded revocation) — an access timestamped at/after the revocation is flagged (`basis:'revoked-at'`) even if the controller placed it BEFORE the withdraw entry.
- **"withhold the withdrawal entirely" — REAL (inherent to a self-maintained log), MITIGATED + documented.** A controller can omit the withdraw entry. New `opts.knownRevokedAt` lets a subject/regulator who holds their own revocation flag a `withheld-withdrawal` (plus any post-revocation access) out-of-band. Completeness of a self-attested log ultimately needs the subject's independent copy — the documented deny-list-propagation residual.
- **"access after the grant EXPIRY not caught" — REAL, FIXED.** The per-access scope check now passes `now:e.at` to `verifyConsent`, so an access after the receipt's `expires_at` is out-of-scope (`verified:false`), not silently compliant.
- **Honest scope (unchanged):** pqconsentflow is verifiable, non-repudiable EVIDENCE the controller commits to — NOT access enforcement, and not a trusted oracle for a fully-malicious controller's self-report. What it guarantees: the controller can't deny what it signed, can't fabricate the subject's consent/withdrawal, and — given the subject's revocation — can't hide a violation. (`verified` vs `compliant` are deliberately distinct so an authentic-but-violating log IS the evidence.)

## pqgate policy-rollback hardening (1 Jul) — DeepSeek (highest-blast-radius engine)
Dedicated adversarial pass on the CI/CD admission gate (it decides what reaches production). DeepSeek found 1 real gap; fixed + re-greened (pqgate 26/26; full gate green):
- **"policy rollback / silent floor-weakening" — REAL, FIXED.** verifyDecision re-evaluated against ANY authority-signed policy whose `policy_id` matched the decision — it never bound to the CURRENT policy. An attacker replaying an OLD decision + its OLD (still-signed) policy could revive a weaker floor (lower minVersion / empty revoked / `require_artifact:false`) and pass verification. Fixed with `opts.trustedPolicyId`: a consumer pins its current `policy_id` (from its registry) and verifyDecision REJECTS a decision whose `policy_id` ≠ the pin (a `policy_pinned` flag signals it) — the same anti-rollback discipline as pqadmit `minVersion` + pqmonitor `minSeq`. Also strengthened the recompute to bind `app`/`version`/`cert_level` (not just `allow` + `cert_id`). Residual (documented): without the pin, verifyDecision proves authenticity + correct evaluation under a signed policy but NOT freshness — pinning is the consumer's anti-rollback obligation.
- **Confirmed sound (no break found):** the decision signature covers every field (tamper → FAILS); artifact binding via the cert `artifact_digest` (swapped binary → recompute DENY); same-call policy-swap caught by the `policy_id` match; `admit()` fail-dangerous under an unverified policy.
