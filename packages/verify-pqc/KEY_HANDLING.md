# Key Handling, RNG, Zeroization & Supply Chain (audit input)

Documents the SDK's key-management posture, randomness sourcing, memory-hygiene limits, and dependency/supply-chain
model — the assumptions an auditor must check against the deployment runtime. Reference implementation, **unaudited**.

## 1. Key ownership model — caller-owned, SDK-stateless
The SDK is a **library of verifiers + signers**, not a key store. With one exception (`pqvault`, §4), every function
takes keys as explicit arguments and holds no key state:
- **Signers** take a secret key (e.g. `ml_dsa87` secret, an identity object) as a parameter; the SDK never persists it.
- **Verifiers** take a *public* key (often a **pinned** trusted key) as a parameter; trust is the caller's pin, not a
  vendor default (verifiers without a pin return *validity*, not *trust* — labelled as such, e.g. `pqverify-api`).
- The **caller owns** key generation, storage, transport, rotation cadence, and destruction. The SDK provides the
  cryptographic operations and the *evidence* that an operation occurred.

## 2. Key generation & RNG assumptions
- Keygen is delegated to the audited primitives: `ml_kem1024.keygen` / `ml_dsa87.keygen` / `slh_dsa_*.keygen`
  (`@noble/post-quantum`), `x25519` / `ed25519` (`@noble/curves`). The SDK adds **no custom keygen or RNG**.
- Randomness comes from `@noble/hashes` `randomBytes`, which uses the **host platform CSPRNG**
  (`crypto.getRandomValues` in browsers / `node:crypto` randomness on the server). **Assumption the auditor must
  validate in the target runtime:** the CSPRNG is properly seeded and not a weakened/test build.
- **Deterministic, seeded keygen** (`keygen(new Uint8Array(32).fill(n))`) appears throughout the **tests** for
  reproducibility — it MUST NOT be used to generate production keys. (The domain-separation/tamper/KAT harnesses use
  fixed seeds by design; production callers pass CSPRNG-generated material.)
- ML-DSA signing: the reference uses the standard (hedged/randomized or context-deterministic) `sign` path of the
  primitive; `extraEntropy` is exercised only in KAT determinism tests. No nonce is generated by SDK code for signing.

## 3. Zeroization — explicit JS-runtime limitation (HONEST)
**JavaScript cannot reliably zeroize secret memory.** Strings are immutable, the garbage collector may copy/retain
buffers, and there is no `explicit_bzero`/`mlock` equivalent. Therefore:
- Secret keys held in JS memory (even transiently) **may persist** until GC and **cannot be guaranteed wiped**. This is
  a property of the runtime, not a defect we can fully fix in-library; it is a **standing residual** for any
  high-assurance deployment and a primary item for the side-channel/operational audit.
- Partial mitigations the SDK follows: minimize secret lifetime (operate-then-drop references), prefer `Uint8Array`
  (which *can* be best-effort `.fill(0)`'d) over strings for raw key bytes where the primitive's API allows, and the
  **crypto-shred** pattern in `pqvault` (§4) — destroy the *custody key* to render envelopes unrecoverable, rather than
  trying to wipe plaintext.
- The auditor should assess: any path that serializes/logs a secret; long-lived secret references; and whether a
  constant-time + memory-hygiene hardening pass (the audit's Task 1) is warranted before production.

## 4. The one stateful module — `pqvault` custody
`pqvault` holds an **in-memory** `Map` of sealed entries + a `pqsign` Merkle manifest. Honest limits (also in its
header + `AUDIT_DOSSIER §7`): (a) **custody keys are long-lived by design** (KMS envelope) — destroy = crypto-shred;
(b) the reference has **no at-rest persistence** — a production deployment needs encrypted storage + a **WAL** so
put+manifest-append are atomic; (c) rollback resistance is conditional on a fresh, witness/gossip-checked manifest head
(documented). A keystore-passphrase rotation utility (`rotate_keystore_passphrase.py`) exists for the deploy path.
`polarseek` (the KEM envelope) likewise assumes a properly-managed custody key supplied by the caller/KMS.

## 5. Supply chain & dependency pinning
- Dependencies are the audited `@noble` family only: `@noble/post-quantum`, `@noble/hashes`, `@noble/ciphers`,
  `@noble/curves` — exact versions recorded in **`sbom.cdx.json`** (CycloneDX) and **`REPRODUCIBILITY.md`** (offline,
  deterministic build). No transitive runtime deps beyond these. *(Known item flagged in the SBOM: a `@noble/hashes`
  1.x↔2.x version-skew to reconcile — confirm a single pinned major across all `@noble` packages.)*
- The published verifier package uses a `files` allowlist that **deliberately excludes** the SDK modules (lean
  attack surface); see `PACKAGING.md`. CI publishes on release; npm provenance / a fresh token in CI secrets is the
  owner's supply-chain control (do not embed tokens).
- **Auditor supply-chain checks:** lockfile integrity, that the installed `@noble` artifacts match the SBOM hashes,
  reproducible build from `REPRODUCIBILITY.md`, and no untracked network/runtime dependency.

## 6. What the audit should specifically confirm (key-handling scope)
1. CSPRNG sourcing + seeding in the *actual* deployment runtime (browser vs Node vs edge).
2. No code path logs, serializes, or transmits a secret key (static + dynamic check).
3. The JS-zeroization residual is documented to downstream integrators (it cannot be eliminated in-library).
4. `pqvault`/`polarseek` custody-key lifetime, crypto-shred completeness, and at-rest/WAL story for any persistent deployment.
5. Dependency pinning + provenance (SBOM match, lockfile, single `@noble` major).
